## 执行引擎设计方案

### 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                    前端（浏览器）                          │
│                                                         │
│  ┌──────────┐    ┌──────────────┐    ┌───────────────┐  │
│  │  画布编排  │───▶│  执行控制器   │───▶│  执行状态展示  │  │
│  │ (ReactFlow)│    │ (WorkflowRun)│    │ (进度/日志)   │  │
│  └──────────┘    └──────┬───────┘    └───────────────┘  │
│                         │ POST /api/workflow/run        │
└─────────────────────────┼───────────────────────────────┘
                          │
┌─────────────────────────┼───────────────────────────────┐
│                 后端（Node.js API 服务）                   │
│                         ▼                                │
│  ┌────────────────────────────────────────────┐          │
│  │              Workflow Engine                │          │
│  │                                              │          │
│  │  1. 拓扑排序（确定节点执行顺序）                │          │
│  │  2. 遍历节点，逐节点分发执行                    │          │
│  │  3. 每个节点：                                 │          │
│  │     a. 接收 input（上游 output）               │          │
│  │     b. 执行节点逻辑                            │          │
│  │     c. 生成 output → 传递给下游                │          │
│  │  4. 收集所有节点结果 → 返回前端                  │          │
│  └────────────────────────────────────────────┘          │
│                         │                                │
│         ┌───────────────┼───────────────┐                │
│         ▼               ▼               ▼                │
│    ┌─────────┐   ┌──────────┐   ┌──────────┐            │
│    │ BMad CLI │   │  Qdrant   │   │ Lark CLI │            │
│    │ (子进程)  │   │ (HTTP)   │   │ (子进程)  │            │
│    └─────────┘   └──────────┘   └──────────┘            │
└─────────────────────────────────────────────────────────┘
```

### 数据流（Pipeline 模型）

```typescript
// 每个节点执行时的上下文
interface NodeExecutionContext {
  nodeId: string
  nodeType: NodeType
  // 来自上游节点的输出（管线数据）
  input: Record<string, any>
  // 该节点自身的配置
  config: Record<string, any>
  // 全局上下文（少量共享数据）
  globalContext: Record<string, any>
}

// 每个节点的执行结果
interface NodeExecutionResult {
  nodeId: string
  status: 'success' | 'error' | 'waiting'
  output: Record<string, any>     // 传递给下游
  logs: string[]
  duration: number
  error?: string
}
```

### 节点执行实现

每个节点类型注册自己的执行逻辑：

| 节点类型 | 执行逻辑 | 输入 | 输出 |
|---------|---------|------|------|
| **用户输入** | 直接传递 input 到 output | 用户输入内容 | `{ text, files, urls }` |
| **智能体** | 调用 AI API (OpenAI/Claude) | 上游文本 | `{ response }` |
| **BMad 角色** | 调用 `bmad-method` CLI | 角色配置+输入 | `{ result }` |
| **Lark 文档** | 调用 `lark-cli` CLI | URL+操作类型 | `{ content, success }` |
| **回答节点** | 暂停等待用户回复 | 问题 | `{ reply }` |
| **AI 输出** | 仅展示结果，无执行 | 上游输出 | 同输入 |

### 执行流程（拓扑排序示例）

```
用户输入 ──▶ 智能体 ──▶ BMad角色 ──▶ Lark写入 ──▶ AI输出
                 │
                 └──▶ 回答节点 ──▶ (等待用户) ──▶ Lark读取
```

拓扑排序引擎会按依赖关系（`edges`）确定执行顺序，并行执行无依赖的节点。

### 前端改动

1. **RunNode 按钮** — 点击时弹出「执行此节点」/「从此节点开始执行」
2. **全局运行按钮** — 画布左上角新增「▶ 运行」按钮
3. **执行状态面板** — 运行中的节点高亮 + 底部执行日志
4. **回答节点** — 执行到 Answer 节点时弹出输入框等待用户回复

### 后端改动

新增一个轻量 Node.js 服务（或直接在现有项目中增加 server/ 目录）：

```
server/
├── index.ts              # Express/Fastify 服务入口
├── engine/
│   ├── workflow.ts       # 工作流执行引擎（拓扑排序 + 节点遍历）
│   ├── executor.ts       # 节点执行器工厂
│   └── nodes/            # 各节点执行逻辑
│       ├── userInput.ts
│       ├── agent.ts      # AI API 调用
│       ├── bmad.ts       # BMad CLI 子进程
│       ├── lark.ts       # Lark CLI 子进程
│       └── answer.ts     # 阻塞等待用户
└── routes/
    └── workflow.ts       # API 路由
```

### 建议分期

| 阶段 | 内容 | 产出 |
|------|------|------|
| **Phase 1** | 前端执行器 + Pipeline 数据模型 + 拓扑排序 | 前端可串行执行节点，日志面板 |
| **Phase 2** | 后端 API 服务 + 智能体节点调 AI | 工作流通过后端执行，调用 AI |
| **Phase 3** | BMad CLI / Lark CLI / Qdrant 对接 | 真实集成外部工具 |
| **Phase 4** | Answer 暂停等待 + 全量/单节点切换 | 完整交互体验 |
